案例说明

案例的中所涉及的是一个影片租赁店的小程序，小程序的主要功能（或者说是需求）：打印顾客的租赁信息，租赁信息主要包括租赁费用和租赁积分；租赁费用和积分都和顾客租赁的影片种类和租赁天数息息相关；目前影片租赁店的影片种类有3种：新片、普通片、儿童片。
根据上述描述信息，作为有经验的开发人员，不说思绪万千，脑海里基本上已经有了小程序的一个大概的框架了（…）。我们简单看一下小程序的雏形，来简单了解一下小程序。

小程序的雏形

几个类及类之间的联系、每个类的属性、主要的函数及函数职能，主要对照着案例说明中的所属信息来进行说明。

讲解完小程序可抛出问题，对小程序的看法？
说出几点：整体设计不好（违背迪米特法则）、扩展性差等…

构建测试

构建测试的目的：避免重构过程中修改的代码出现BUG
所构建的测试用例需要有自我检测功能，减少重构时间

开始重构

每个版本的重构都是为了解决已知的不足，首先抛出问题，确定当前版本要解决的问题，然后说明重构方法，最后代码实现出来。

version01

解决statement()方法过长的问题，提炼函数，解决statememt()一个函数包含多个业务逻辑的问题，明确各个函数业务逻辑。

version02

小程序违法了迪米特法则——最少知道原则：一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。
说白了就是不符合面向对象的编程思想。

version03

解决程序扩展性差的问题，并且switch...case...语句的存在仍是个大问题。

  1. 状态模式
    当一个对象的行为取决于它的状态，并且它必须在运行时刻根据状态改变它的行为时，就可以考虑使用状态模式来。

  2. 子父类继承
    作为最后的终极方案，舍弃switch...case...解决扩展问题，只看看现有的实现代码，作为拓展内容；
    个人认为也是可以的，优点是完全舍弃了switch...case...，解决了扩展问题；缺点是上层逻辑需要改动，进行相应的适配。
